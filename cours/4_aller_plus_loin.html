Les dépots en ligne

Que ce soit sur un serveur privée ou sur des serveurs en ligne, vous pouvez utiliser des dépots distants pour avoir un point central où stocker vos sources. Cela ne changera pas le fait que le système est décentralisé du fait de la duplication des sources sur tout les postes qui reproduise le projet, mais au moins vous pourrez accéder à vos sources de nimporte où.

Il existe plusieurs dépôts distants parmis lesquels les plus connus sont :

  • GitHub
  • GitLab
  • BitBucket

... et bien d'autres.
Tous permette la création de dépôts public (accessible à tous) ou privée (uniquement pour les membres invités).

Utiliser un dépôt en ligne

Pour utiliser les dépôts en ligne il faut :

  • Avoir un compte sur ces dépôts
  • Ajouter la source du dépôt distant à notre Git local
  • Et connaitre au moins 3 nouvelles commandes Git

Avoir un compte en ligne

Pour avoir un compte en ligne il suffit de le créer. Et c'est d'une facilité déconcertante.

Ajouter la source du dépôt distant à notre Git local

Pour ajouter le dépot distant à votre git local il faut utiliser une ligne de commande : git remote add MC https://github.com/Nom-De-Votre-Compte/NomDeVotreProjet.git.
La commande dit : Git ajoute (add) un dépot distant (remote) dont le petit nom est MC et qui se trouve à cette URL.

Voilà votre dépôt distant est ajouté. Vous pouverz aussi ajouter une clef SSH pour simplifié la connexion, mais pour cela je vous laisse vous référer au tutoriel présent chez le fournisseur que vous aurez choisis.

Les commandes Git pour travailler avec un dépôt distant

Il existe 3 commandes principale pour travailller avec un dépôt distant sous Git.

  • git clone https://github.com/Nom-De-Votre-Compte/NomDeVotreProjet.git : qui permet de clôner un projet existant sur votre poste. En gros il rappatrie les source et l'arbre Git.
  • git pull : c'est la commande qui permet de mettre à jour votre dépôt local par rapport au dépôt distant.
  • git push : à l'inverse de la précédente, envoie votre code sur le dépôt distant pour le mettre à jour avec vos modifications.

GitFlow

Il n'y a rien de miraculeux dans l'utilisation de ces commandes, mais il faudra être très attentif pour limiter les conflits et les erreurs. Pour ce faire il peut ête très important de définir un GitFlow.

Il en existe plusieurs, mais certaines bases ressortent de tous : la création de branches en fonction de l'avancement du travail.

  • La master : elle représente la version de production de votre projet.
  • La develop : elle représente ce qui est en cour de développement.
  • La feature : c'est plus un dossier qu'une branche, dans lequel se trouverons les différentes branche de fonctionnalité en cour de dévelopement : git branche feature/issue_1.
  • La release : elle prépare ce qui sera livré en production.
  • La hotfix : elle ne sert qu'à corriger les bogues qui seront poussés immédiatement sur la branche master.

Attention cependant, le GitFlow doit être adapté à votre façon de travailler. Qui plus est, il faut bien faire attention à toujours "pull" avant de "commit" vos modifications et de les "push" par la suite. Sans quoi vous risqueriez de pousser des conflits sur le dépot distant, ce qui impacterait tous les participants au projet.
